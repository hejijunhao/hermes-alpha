# 0.2.0 — Hermes Agent Integration

**2026-03-09**

Implementation notes for wiring the Hermes agent into the WebSocket pipeline, replacing the echo stub.

---

## What Was Done

### 1. Server — `server/main.py`

**Replaced the echo stub with a live Hermes agent session.**

- Imports `AIAgent` from `run_agent` (the Hermes package's main module).
- Each WebSocket connection gets its own `AIAgent` instance via `_create_agent()`. This is necessary because `AIAgent` is not thread-safe — it holds internal conversation state.
- Uses `agent.run_conversation()` (not `chat()`) to support multi-turn conversation. The `conversation_history` list is carried across messages within the same WebSocket session, so the agent remembers context.
- `run_conversation()` is synchronous/blocking, so it runs in a thread pool via `asyncio.run_in_executor()` to avoid blocking the event loop.
- Sends a `PROCESSING...` system message before each agent call so the frontend can show a waiting state.
- Catches and forwards agent errors as system messages instead of crashing the WebSocket.

**Configuration (via environment):**

| Variable | Default | Purpose |
|---|---|---|
| `LLM_MODEL` | `nousresearch/hermes-3-llama-3.1-405b` | OpenRouter model identifier |
| `HERMES_MAX_ITERATIONS` | `10` | Max tool-calling loop iterations per message |

**Agent flags chosen:**

- `quiet_mode=True` — suppresses CLI spinners/rich output that would pollute server logs.
- `skip_context_files=True` — prevents the agent from loading `AGENTS.md` from the working directory (irrelevant in a web context).
- `skip_memory=True` — disables persistent cross-session memory. Each WebSocket session starts fresh. Can be flipped later if we want persistent user memory.
- `disabled_toolsets=["terminal"]` — prevents the agent from executing shell commands on the server. Security boundary.

### 2. Dependencies — `server/requirements.txt`

Added:
- `python-dotenv==1.1.*` — loads `.env` file into `os.environ` at startup.
- `hermes-agent[all] @ git+https://github.com/NousResearch/hermes-agent.git` — installs Hermes and all optional toolsets from the main branch.

### 3. Docker — `Dockerfile`

Added `git` installation step. Required because `pip install` needs to clone the Hermes repo from GitHub. Uses `--no-install-recommends` and cleans up apt lists to keep the image lean.

### 4. Frontend — `frontend/main.js`

Added processing state management:
- When the server sends `PROCESSING...`, a pulsing system message appears and input is disabled.
- When the agent's reply (or an error) arrives, the processing indicator is removed and input re-enables.
- Prevents sending new messages while the agent is thinking (guards against queueing).

### 5. Frontend — `frontend/style.css`

Added `.msg.system.processing` class with a pulse animation to visually distinguish the transient "thinking" state from permanent system messages.

### 6. Environment — `.env.example`

Added `LLM_MODEL` and `HERMES_MAX_ITERATIONS` as commented-out optional config.

## Architecture After This Change

```
Browser ──WebSocket──► FastAPI ──thread──► AIAgent ──HTTP──► OpenRouter API
                                              │                   │
                                    conversation_history     LLM inference
                                    (in-memory, per session)  (remote GPU)
```

- **No GPU required** on the server. Hermes is a client-side orchestrator; all inference happens at the API provider.
- **One agent per connection.** Thread-safe by isolation — no shared mutable state.
- **Stateless across sessions.** Refreshing the page starts a new conversation. Persistent memory is disabled (`skip_memory=True`).

---

## Design Decisions

### Why `run_conversation()` over `chat()`?

`chat()` returns just the response string. `run_conversation()` returns the full message history, which we feed back on the next turn. This gives the agent multi-turn context without us manually constructing OpenAI-format message lists.

### Why disable terminal tools?

The agent has built-in terminal/shell toolsets. On a shared server, letting an AI agent run arbitrary commands is a security risk. We disable the `terminal` toolset and can selectively enable safer ones (like `web`) later.

### Why `run_in_executor` instead of an async agent?

Hermes's `AIAgent` is synchronous — it blocks while making API calls and running tool loops. FastAPI's event loop would freeze if we called it directly. `run_in_executor` offloads the blocking call to a thread pool, keeping the event loop responsive for other connections.

### Why one agent per WebSocket?

`AIAgent` maintains internal state (iteration counters, tool sessions). Sharing one instance across connections would cause race conditions. Creating a fresh instance per connection is cheap — the heavy work is the remote API calls, not the Python object.

## What's Not Done Yet

- **Streaming tokens** — responses arrive as one block after the full tool loop completes. The agent doesn't expose a public streaming API; we'd need to hook into its internal `_run_codex_stream` or use `step_callback` for intermediate updates.
- **Persistent memory** — `skip_memory=True` means each session is stateless. Enabling Hermes's built-in memory system would give cross-session recall.
- **Conversation persistence** — no server-side storage. History lives only in the WebSocket session's memory.
- **Model selection UI** — model is set via env var. Could expose a `/settings` endpoint or UI picker.
- **Rate limiting / auth** — the WebSocket endpoint is open. Fine for personal use, needs gating for public deployment.
